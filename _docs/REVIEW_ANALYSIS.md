## 1. 設計矛盾・曖昧点

### 1.1 RLSとAPI層の認可が二重定義されている

- タスク#71でRLSポリシーを「全テーブルRLS有効化」「6パターン定義」としている
- タスク#140でAPI基盤に`withAuth()` JWT検証ミドルウェアを定義している
- タスク#85でNext.js middleware による認証ガードも定義している

**問題**: 認可ロジックが3層（RLS / API middleware / Next.js middleware）に分散し、どの層がどの責務を持つか不明確。例えば「非管理者の /admin アクセス拒否」はNext.js middlewareだが、「他人のプロフィールを更新できない」はRLSなのかAPI層なのか判断基準がない。実装者によって判断が割れ、セキュリティホールが生まれるリスクがある。

**推奨**: 各層の責務を明文化すること。例: Next.js middleware = ルーティングレベルの認証・リダイレクト、API middleware = リクエスト認証とJWT検証、RLS = データアクセスレベルの最終防壁（defense in depth）。

### 1.2 messagesテーブルがPhase 2なのにPhase 1でスキーマ定義すべき?

- タスク#41: messagesテーブルは「Phase 2で使用」と明記
- タスク#113: MessageThread コンポーネントも Phase 2
- タスク#148: メッセージ系APIもPhase 2

**問題**: 矛盾ではないが曖昧。Phase 1でスキーマだけ作るのか、完全にPhase 2まで先送りか。マイグレーション管理の観点からPhase 1で空テーブルだけ作ることのメリットがない。

### 1.3 `contact_info` の可視性制御が曖昧

- タスク#39: `contact_info` は `TEXT nullable: LINE ID/メール等`
- タスク#111: ProfileModalで「contact_info表示（コネクション成立後）」
- タスク#177: 同様

**問題**: この可視性制御をどこで行うか未定義。RLSではカラム単位の制御ができないため、RLSだけでは実現不可能。API層で条件分岐してレスポンスから除外するか、DB関数でフィルタリングするか、フロントエンドで非表示にするだけかで、セキュリティレベルが大きく異なる。フロントエンド非表示のみだとAPIを直接叩けば見える。

**推奨**: API層でconnectionステータスを確認し、未接続ユーザーには`contact_info`をnullとして返却する設計を明示すべき。

### 1.4 `notifications.actions` JSONBの「ホワイトリスト制御」の仕様が不明

- タスク#42: `actions(JSONB: allowedActionsホワイトリスト制御)`
- タスク#112: `actions ホワイトリスト実行`

**問題**: ホワイトリストの定義場所、アクションの種類、フロントエンドでの実行メカニズムが一切記述されていない。JSONBにアクション定義を入れるということは、サーバーから任意のアクションをクライアントに指示するアーキテクチャだが、これはXSSリスクに直結しうる（例: actionsにURLやコールバックを含める場合）。

**推奨**: actionsのスキーマ（型定義）、許可されるアクション名の列挙、フロントエンドでのディスパッチ方法を設計書で明示すべき。

### 1.5 `matching_scores_v2` の「viewer/target」と双方向性の関係

- タスク#49: `UNIQUE(viewer_id, target_id)` -- AからBとBからAは別レコード
- タスク#169: 「全アクティブペアの双方向」バッチ処理

**問題**: nユーザーに対して最大 n*(n-1) レコードが生成される。これが意図的かどうか。双方向スコアが本質的に非対称（AのニーズをBが満たすか vs BのニーズをAが満たすか）であることは理解できるが、ストレージとバッチ処理時間の爆発に直結する。後述のスケーラビリティ懸念参照。

### 1.6 tl;dv webhook と Cron Job の役割分担

- タスク#155: `tldv-webhook` Edge Function（署名検証 + transcripts更新）
- タスク#158: Cron Job（tl;dv REST呼出し、新規検出 → transcripts + participants UPSERT）

**問題**: webhookでリアルタイムに受け取るなら、なぜCron Jobでもポーリングするのか。冗長性のためか、webhookでは取得できない情報があるのか、フォールバックなのか不明。両方が同じデータを書き込む場合、競合状態（race condition）が発生する。`tldv_meeting_id` のUNIQUE制約で重複挿入は防げるが、ステータス更新の競合は防げない。

**推奨**: Cron Job = webhookの取りこぼし補完（reconciliation）と明記し、UPSERTの競合解決ルール（例: statusが進んでいる方を優先）を定義すべき。

### 1.7 score_reasons 生成のタイミングと方法

- タスク#174: 「Claude Sonnet 4.6 で理由テキスト」

**問題**: 全ペアのスコア理由をClaude APIで生成するのか。n*(n-1) ペアに対して毎回Claude APIを呼ぶのはコスト的にも速度的にも非現実的。差分（is_stale=trueのみ）だとしても、1ユーザーのプロフィール更新で全対ペアがstaleになる。テンプレートベースの理由生成で十分ではないか。

## 2. 過剰設計の指摘

### 2.1 Phase 1 でダークモードトークン定義（タスク#23）

設計書には「切替UIはPhase2」とあるが、「全カラーのダーク版をPhase1で定義」と明記。

**指摘**: ダークモードの利用者がいないPhase 1で全トークンのダーク版を定義する工数は、プレローンチのMVPとして過剰。ライトモードの配色すら使ってみて調整が必要になるのに、ダーク版を先行定義するとダブルメンテナンスが発生する。Phase 2で切替UIと同時に定義すれば十分。

### 2.2 Phase 1 のデザインシステムタスク数 (18タスク)

タスク#18〜#35: 設計原則、競合調査、カラー策定、タイポグラフィ、スペーシング、シャドウ、コピーライティングガイド、モーション定義、ビジュアルアセット。

**指摘**: MVPで18タスクをデザインシステムに割くのは過剰。特に以下は Phase 1 で不要:
- タスク#19（競合・参考調査）: ローンチ前に完了しているべきプロダクト判断であり、構築タスクではない
- タスク#30（UIコピーガイドライン）: 全UIコピーのガイドラインを事前策定するより、実装しながら統一する方が現実的
- タスク#33（モーション定義）: 「動く要素5箇所/ページ以下」の制約はPhase 1では自然に満たされる。prefers-reduced-motion対応は個別実装時で十分

### 2.3 Phase 1 で profile_views のデータ蓄積（タスク#55, #139, #142）

- タスク#55: 「データ蓄積Phase1、表示UIはPhase2」
- タスク#139: IntersectionObserver + 3秒経過で POST
- タスク#142: APIエンドポイント実装

**指摘**: 表示UIがPhase 2なのに、データ蓄積ロジック（IntersectionObserver、3秒タイマー、API）をPhase 1で構築する合理性が薄い。ユーザーに見えない機能に工数を使うより、コアフローの品質向上に充てるべき。Phase 2でUI込みで一括実装した方が、仕様変更（例: 蓄積ルールの変更）にも柔軟。

### 2.4 login_sessions のPhase 1記録（タスク#87, #61）

- タスク#87: 「ログイン成功時にlogin_sessions記録（device/browser/ip）」Phase 1
- タスク#61: テーブル定義はPhase 2、UI は Phase 3

**問題**: テーブル定義がPhase 2なのに記録ロジックがPhase 1は矛盾。仮にPhase 1でテーブルも作るとしても、UIがPhase 3まで存在しないデータを蓄積するだけの機能。セキュリティ監査用途ならログで十分。

### 2.5 bookmarks のメモ機能（タスク#56, #112）

**指摘**: ブックマーク自体はシンプルだが、メモ（note）フィールドの追加はPhase 1で本当に使われるか疑問。ブックマーク=フラグで十分なMVPフェーズ。メモのUIも含めて工数が増える。

### 2.6 5軸スコアリングの複雑さ

タスク#171〜#173で5軸（needs_fulfillment 35% / skill_complementarity 25% / communication_compatibility 15% / engagement_quality 15% / interaction_history 10%）を定義。

**指摘**: Phase 1のローンチ時点では大半のユーザーがトランスクリプトデータを持っていない。タスク#170のフォールバック（transcript系3軸=50中立）が実質的にほとんどのケースで適用される。つまりPhase 1では5軸のうち2軸（needs/skill）がプロフィールベースの簡易評価、3軸が固定値50になる。5軸の精緻な設計をPhase 1で完成させる意味は薄い。2軸（needs+skill）のシンプルなスコアリングから始め、データが溜まったPhase 2で5軸に拡張する方が現実的。

## 3. 技術選定の妥当性レビュー

### 3.1 Supabase RLS vs API層での認可

**設計書の方針**: RLSで全テーブル保護（タスク#71）+ API層でwithAuth() JWT検証（タスク#140）

**評価**: Defense in depth として両方持つのは正しいが、以下の懸念がある:

1. **RLSのデバッグ困難性**: RLSポリシーはSQLで記述するためテストが難しく、誤ったポリシーによるデータ漏洩を発見しにくい。特に6パターンの複雑なポリシーは相互干渉のリスクがある
2. **パフォーマンス影響**: 全クエリにRLSのWHERE句が付加される。`matching_scores_v2`のような大量レコードテーブルでのRLSは、インデックスとの整合性を慎重に設計しないとクエリプランが劣化する
3. **Supabaseクライアント直接アクセス**: フロントエンドから`supabase-js`で直接DBにアクセスする場合はRLSが必須だが、全アクセスをAPI Route Handler経由にするならRLSは二重防壁として機能する。設計書はどちらのパターンを主とするか明示していない

**推奨**: 読み取り系はsupabase-js直接（RLS依存）、書き込み系はAPI Route Handler経由（API層で検証 + RLSで防壁）という方針を明記すべき。

### 3.2 Zustand + TanStack Query の組み合わせ

**設計書の方針**: Zustand（UIStore/FilterStore）+ TanStack Query（データ取得/キャッシュ）（タスク#93〜#97）

**評価**: 妥当な選定。

- TanStack Queryがサーバー状態（プロフィール、コネクション、通知等）を担当
- Zustandがクライアント状態（モーダル開閉、フィルター条件）を担当

責務分離が明確で、一般的なベストプラクティスに沿っている。ただし以下に注意:

1. **FilterStoreとURL stateの二重管理**: タスク#176で`?highlight=`クエリパラメータを使っている。フィルター条件もURLに持たせるなら、ZustandのFilterStoreとURLが二重の真実の源になる。`nuqs`やTanStack Routerのsearch params管理を検討すべき
2. **Realtimeとの統合**: タスク#99〜#101でRealtime購読を定義しているが、Realtimeイベントで TanStack Queryのキャッシュをinvalidateする連携パターンを明示すべき

### 3.3 Render Worker vs Supabase Edge Function の使い分け

**設計書の方針**:
- Render Worker: Claude API呼び出し（トランスクリプト分析）、双方向スコアリングバッチ（タスク#162, #169）
- Supabase Edge Function: tl;dv webhook、TimeRex webhook（タスク#155）
- Render Cron Job: tl;dv定期取得（タスク#158）

**評価**: おおむね合理的だが、使い分けの基準が暗黙的。

- Edge Function = webhook受信（低レイテンシ、短時間処理）に適切
- Render Worker = 長時間バッチ処理に適切（Supabase Edge Functionは実行時間制限があるため）

**懸念点**:
1. **Edge FunctionからRender Workerへのジョブ投入方法が未定義**: tl;dv webhookで新しいtranscriptを受信した後、分析処理をどうトリガーするか。Edge Function → Render WorkerへのHTTPリクエスト? キューイング? 設計書に記載なし
2. **Render Workerのジョブキュー**: タスク#164でWorker起動時リカバリ（status=analyzingを再キュー）は言及しているが、通常のジョブキューイングメカニズム（BullMQ? PostgreSQLベースのキュー?）が未定義
3. **Supabase Edge FunctionのDeno制約**: Deno/TSで記述するが、使用できるnpmパッケージに制限がある。署名検証ライブラリの動作確認が必要

**推奨**: ジョブキューの実装方式（PostgreSQLのLISTEN/NOTIFY、またはポーリング）を明示し、Edge Function → Worker の連携フローを設計書に追記すべき。

### 3.4 tl;dv API の制約・レート制限への対策

**設計書の記載**: タスク#12で「API Key取得」、タスク#158でCron（JST 9/12/15/18の4回/日）。

**懸念点**:
1. **レート制限の調査結果が未記載**: tl;dv APIのレート制限（requests/minute等）が設計書にない。Cron 4回/日は控えめだが、1回のCron実行で何件のtranscriptをフェッチするか、ページネーションが必要か不明
2. **APIの可用性**: tl;dv APIがダウンした場合のリトライ戦略がCron側にない（タスク#163のリトライはClaude API呼び出し側のみ）
3. **transcript全文取得のサイズ**: `full_text`カラムにトランスクリプト全文を格納する設計だが、長時間会議のトランスクリプトは数十KBになりうる。Claude APIのコンテキストウィンドウ制限との整合性を確認すべき
4. **tl;dv APIのデータ形式**: スピーカー情報がどの程度構造化されているか（スピーカー名、メールアドレス等）によって、タスク#159の多段階紐付けアルゴリズムの実現可能性が変わる

## 4. セキュリティ懸念

### 4.1 RLS設計

1. **service_roleキーの管理**: タスク#71で「service_role」パターンを定義。service_roleキーはRLSをバイパスするため、フロントエンドに絶対に露出してはならない。`NEXT_PUBLIC_`プレフィックスの環境変数分類（タスク#16）で明確に除外されていることを確認すべき
2. **SECURITY DEFINER関数のリスク**: タスク#67でポイント操作関数に`SECURITY DEFINER + REVOKE`を適用。SECURITY DEFINER関数はRLSを無視して定義者の権限で実行されるため、入力バリデーションの不備が直接的な権限昇格につながる。しかしZodバリデーション（タスク#183）はPhase 2
3. **JSONBカラムのインジェクション**: `actions`, `communication_traits`, `score_reasons`等のJSONBカラムに格納されたデータがフロントエンドでそのままレンダリングされると、Stored XSSのリスクがある

### 4.2 JWT検証

1. **Supabase JWTの検証方法が未定義**: タスク#140の`withAuth()`がSupabaseのJWTをどう検証するか。`supabase.auth.getUser()`（サーバーサイド検証、毎回Supabase Auth APIを叩く）か、JWTのローカル検証（`jsonwebtoken`ライブラリ + JWT Secret）かで、セキュリティレベルとパフォーマンスが異なる
2. **Edge FunctionでのJWT検証**: タスク#155のwebhook Edge Functionは外部サービスからの呼び出しなので、JWTではなく署名検証が正しいが、他のEdge Functionを追加した場合の認証方式が未定義

### 4.3 Facebook OAuth

1. **scope（権限範囲）が未定義**: `signInWithOAuth({ provider: 'facebook' })`のデフォルトscopeは`email`だが、プロフィール情報（name等）も必要ならscope追加が必要。取得できる情報とuser_profilesへのマッピングが未定義
2. **Facebook App Reviewの要件**: 本番環境で一般ユーザーがFacebook Loginを使うには、FacebookのApp Reviewプロセスを通す必要がある（2024年以降の要件）。設計書にこのプロセスが含まれていない
3. **既存メールアカウントとの統合**: EmailでSignUp済みのユーザーが同じメールアドレスのFacebookアカウントでOAuthログインした場合の挙動（アカウントリンク or 別アカウント作成 or エラー）が未定義。Supabaseのデフォルト設定による暗黙の動作に依存するのは危険

### 4.4 Webhook署名検証

1. **tl;dv署名検証の仕様が不明**: タスク#187で「tl;dv署名検証（Phase1）」とあるが、tl;dvがどの署名方式を使うか（HMAC-SHA256? カスタムヘッダー?）が未記載。tl;dv APIドキュメントの確認が必要
2. **リプレイ攻撃対策**: 署名検証だけではリプレイ攻撃を防げない。タイムスタンプ検証（例: 5分以内のリクエストのみ許可）が必要だが未言及
3. **webhook受信エンドポイントのレート制限**: 署名検証はするが、不正なリクエストの大量送信によるDDoSは防げない。Edge Functionレベルでのレート制限が未定義

## 5. スケーラビリティ懸念

### 5.1 双方向マッチングのO(n^2)問題

**設計書の記載**: タスク#169「全アクティブペアの双方向。差分（is_stale=trueのみ）」

**分析**:
- n人のアクティブユーザーに対し、最大 n*(n-1) ペアのスコア計算が必要
- 100人: 9,900ペア -- 問題なし
- 1,000人: 999,000ペア -- Claude API呼び出しを含むなら非現実的
- 10,000人: 約1億ペア -- 格納すら困難

**is_stale差分処理の限界**: 1人のプロフィール更新で、その人に関連する全ペア（n-1件）がstaleになる。頻繁なプロフィール更新があるとstaleの割合が常に高くなり、差分処理の意味が薄れる。

**現実的な対策案**:
1. **計算対象の制限**: 全ペアではなく、同業種・類似スキルなど事前フィルタリングで候補を絞る（例: 上位100人のみ計算）
2. **オンデマンド計算**: バッチ処理ではなく、ユーザーがマッチング画面を開いた時にリアルタイム計算（キャッシュあり）
3. **score_reasonsのテンプレート化**: タスク#174のClaude API呼び出しをテンプレートベースに変更し、APIコストを削減
4. **段階的計算**: まず軽量な2軸（needs/skill）でフィルタリングし、上位候補のみ5軸フルスコアを計算

**推奨**: Phase 1ローンチ時の想定ユーザー数を明記し、その規模で成立する計算方式を選択すべき。100人規模なら全ペアでも成立するが、スケール時の移行パスを設計書に記載しておくこと。

### 5.2 Claude API呼び出しのコストとスループット

**設計書の記載**:
- トランスクリプト分析（タスク#162）: 発言者ごとに構造化抽出
- score_reasons生成（タスク#174）: 全staleペアに理由テキスト生成

**コスト試算**（概算）:
- 1会議のトランスクリプト分析: 入力 ~5,000トークン、出力 ~2,000トークン x 参加者数
- score_reasons: 入力 ~1,000トークン、出力 ~500トークン x ペア数
- Claude Sonnet 4.6の料金でペア数が増えると急速にコスト増

**対策が未記載**: APIコストの予算上限、月次コスト見積もり、コスト超過時のフォールバック（例: 理由生成をスキップ）が設計書にない。

### 5.3 Realtime接続数

**設計書の記載**: タスク#99〜#101で全認証ユーザーがRealtime購読。

**懸念**: Supabase Proプランの同時Realtime接続数には制限がある（デフォルト500）。ユーザー数がこれを超えた場合の対策が未記載。通知のポーリングへのフォールバック、またはRealtimeチャネルの設計（ユーザーごとの個別チャネル vs 共有チャネル + フィルタリング）の詳細が必要。

### 5.4 Supabase Storage のアバター/カバー画像

**設計書の記載**: タスク#72でアバター2MB、カバー5MB。

**懸念**: 画像のリサイズ処理が未定義。ユーザーが2MBの高解像度画像をアップロードし、一覧画面で数十人分のアバターを表示すると、転送量が膨大になる。Supabase Storageの画像変換機能（Proプランで利用可能）を使うか、アップロード時にサーバーサイドリサイズするかの方針が必要。Cloudflare Image OptimizationはPhase 2（タスク#189）のため、Phase 1では対策がない。

## 総括

### 高リスク事項（Phase 1リリース前に解決必須）

| # | 事項 | 理由 |
|---|------|------|
| 1 | contact_info の可視性制御をAPI層で実装 | フロントエンド制御のみだとプライバシー侵害 |
